# Safe City Backend Context

## Architecture Overview
- **Framework**: FastAPI (Python)
- **Database**: PostgreSQL with SQLAlchemy ORM
- **In-Memory Store**: Redis (for locking, caching, and background tasks)
- **Migrations**: Alembic
- **Deployment**: Docker / Docker Compose

## Project Structure
- `app/`
  - `api/`: Endpoint handlers organized by domain (`user.py`, `guard.py`, `emergency.py`, etc.).
    - `deps.py`: Dependency injection (auth, db sessions).
  - `models/`: SQLAlchemy models (User, Guard, Emergency, SecurityCompany, etc.).
  - `schemas/`: Pydantic models for request/response validation.
  - `services/`: Core business logic layer.
    - `dispatch.py`: Logic for assigning guards to emergencies.
    - `routing.py`: Pathfinding and distance calculations.
    - `emergency.py`: Emergency call lifecycle management.
    - `geocoding.py`: Location-to-address and vice versa.
  - `core/`: Global configuration (`config.py`), security (`security.py`), and init logic.
- `alembic/`: Database migration scripts.
- `docker-compose.yml`: Infrastructure (Backend, Postgres, Redis, OSRM).

## Key Entities & Roles
1.  **User**: Standard customer. Must be `ACTIVE` and often requires an active subscription (`require_subscription`) to trigger emergencies.
2.  **Guard**: Security responder. Must be `active` to receive assignments.
3.  **Security Company**: Managed by **Company Admins**.
4.  **Company Admin**: Manages companies and guards. Roles include `owner` (for privileged actions).
5.  **Emergency Call**: The central entity connecting users and guards.
6.  **Review**: Post-emergency feedback.
7.  **Payment/Subscription**: User billing logic.

## Authentication & Authorization
JWT-based authentication managed in `app/core/security.py`.
- **Dependencies** (`app/api/deps.py`):
    - `get_current_user`: Validates user tokens.
    - `get_current_guard`: Validates guard tokens.
    - `get_current_admin`: Validates company admin tokens.
    - `require_subscription`: Ensures the user has an active billing plan.

## Core Logic
-   **Assignment Logic**: Located in `app/services/dispatch.py`. Uses radial searches and distance metrics to find the best-fit guard.
-   **Routing**: Relies on an OSRM instance (as seen in `docker-compose.yml`) for real-time navigation data.
